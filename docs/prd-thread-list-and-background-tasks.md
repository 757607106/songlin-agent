# PRD：对话列表状态与后台任务支持



## 1. 背景与目标
- 目标：在对话应用中提供“对话列表状态可视化 + 后台任务可持续运行”的能力。
- 约束：前端可随时断连/刷新，任务在后端继续执行并可恢复。
- 范围：对话列表、状态筛选、状态提示、任务进度、人工中断审批。

## 2. 用户需求
- 用户能快速查看所有对话的运行状态（空闲/忙碌/中断/错误）。
- 用户能筛选某一状态的对话列表。
- 用户在刷新页面或切换页面后能继续看到任务运行进度。
- 用户在中断场景下可进行批准/编辑/拒绝操作。

## 3. 需求清单

### 3.1 功能需求
- F1：对话列表按状态展示彩色标识。
- F2：对话列表支持按状态过滤。
- F3：对话列表支持时间分组（今天/昨天/本周/更早）。
- F4：中断状态需单独分组为“需要关注”。
- F5：对话状态应在以下事件后刷新：
  - 新消息发送
  - 流结束/失败
  - 新线程创建
  - 中断恢复或人工审批后
- F6：对话任务在后台持续执行（前端可断连）。
- F7：任务进度支持独立显示（任务列表/摘要）。
- F8：中断时提供操作面板（批准/编辑/拒绝）。

### 3.2 非功能需求
- N1：列表分页加载，避免一次性拉全量数据。
- N2：断网/重连时能恢复线程状态。
- N3：状态展示与业务逻辑解耦（后端为单一状态源）。

## 4. 业务逻辑（实现逻辑）

### 4.1 状态来源
- 状态来自后端线程列表接口（线程搜索）。
- 前端不自行计算状态，只做展示与过滤。

### 4.2 状态刷新机制
- 对话执行流的生命周期事件触发列表刷新：
  - 流完成、流错误、新线程创建
  - 用户发送消息或继续执行
  - 中断恢复/审批后

### 4.3 状态分组与筛选
- 逻辑先按“中断”优先分组。
- 其余按更新时间分组（今天/昨天/本周/更早）。
- 筛选条件由前端传给后端，后端返回对应状态线程。

### 4.4 后台任务可持续运行
- 任务状态存储在后端线程中。
- 前端通过线程 ID 重新连接执行流。
- 只要线程存在，任务可持续运行，不依赖前端连接。

### 4.5 任务进度
- 任务列表来自执行流的状态值（例如 todos）。
- 任务状态：待处理 / 进行中 / 已完成。
- UI 可显示任务摘要或完整列表。

### 4.6 中断审批
- 中断产生时，前端读取中断结构数据。
- 按“允许的决策”展示按钮：批准/编辑/拒绝。
- 用户提交决策后恢复执行流。

## 5. 当前项目技术架构（概览）

### 5.1 架构层次
- **UI 层**：对话列表、对话窗口、任务/文件侧边栏、工具调用与中断审批组件。
- **状态层**：
  - 列表状态由 `useThreads` 拉取并缓存（SWR）。
  - 对话执行由 `useStream` 管理（LangGraph SDK）。
- **数据层**：LangGraph SDK `Client` 与线程搜索/执行流接口。

### 5.2 主要模块职责
- **线程列表模块**：拉取线程、过滤、分页、分组、显示状态。
- **执行流模块**：提交消息、继续执行、恢复中断、停止执行。
- **任务模块**：展示执行流中的任务列表（todos）。
- **中断模块**：渲染审批 UI 并提交用户决策。

### 5.3 关键数据结构
- 线程状态：`idle | busy | interrupted | error`。
- 任务状态：`pending | in_progress | completed`。
- 中断决策：`approve | edit | reject`。

## 6. 复刻到其他架构的适配原则
- **状态单一来源**：线程状态应由后端管理，前端只展示。
- **可恢复执行**：任何任务应能通过 `threadId` 重新连接恢复。
- **可观测进度**：任务进度应从执行流状态中派生。
- **中断可决策**：当执行流需要人工输入时，必须提供决策入口。

## 7. 不在本 PRD 范围
- 对话内容的具体展示样式。
- 模型能力或业务知识逻辑。
- 文件系统内容的格式规范。
